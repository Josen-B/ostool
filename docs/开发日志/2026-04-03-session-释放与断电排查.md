# 2026-04-03 Session 释放与断电排查

## 背景

本次排查的直接现象是：

- `ostool-server` session 退出时，开发板没有稳定断电
- `ostool board connect -b orangepi5plus` 在客户端按 `Ctrl+A`、`x` 退出后，偶发断电失败

相关环境：

- server 配置文件：`.ostool-server.toml`
- 板卡配置：`.ostool-server/boards/orangepi5plus-1.toml`
- 电源管理类型：`zhongsheng_relay`

## 先确认的事实

### 1. 这块板不是命令式断电

`orangepi5plus-1` 的电源配置不是 `custom`，而是：

```toml
[power_management]
kind = "zhongsheng_relay"
serial_port = "/dev/ttyUSB0"
```

这意味着问题不是 `power_off_cmd` 写错，而是：

- 是否真的触发了 release / power-off
- 继电器 Modbus 写线圈是否被调用
- 是否有重复 power-off 导致超时

### 2. 中盛继电器的关键协议参数

根据 `docs/自动上电/中盛数字量输入输出操作手册.md`：

- 默认地址：`1`
- 默认波特率：`38400`
- 数据位：`8`
- 停止位：`1`
- 校验：`无`
- 通道 1 线圈地址：`0000H`

当前代码中对应常量为：

- `Slave ID = 1`
- `baud rate = 38400`
- `coil address = 0`

与手册一致。

## 调试过程中的关键结论

### 1. 手动 power-off API 是成功的

用 API 直接创建 session 并立即调用：

```bash
curl -sS -H 'Content-Type: application/json' \
  -X POST http://127.0.0.1:2999/api/v1/sessions \
  -d '{"board_type":"orangepi5plus","required_tags":[],"client_name":"debug-cli"}'

curl -sS -X POST \
  http://127.0.0.1:2999/api/v1/sessions/<session_id>/board/power-off
```

server 返回：

```json
{"ok":true,"message":"executed Zhongsheng relay power-off via /dev/ttyUSB0"}
```

这说明：

- 继电器串口 `/dev/ttyUSB0` 可打开
- Modbus 写线圈至少在单次调用场景下可成功

### 2. 问题真正出在“重复释放”

在早期实现中，客户端 `Ctrl+A+x` 退出后会有两条路径都去执行 `power-off`：

1. serial websocket 关闭后的自动清理路径
2. 客户端 `session.release()` 发出的 `DELETE /sessions/{id}` 路径

现场日志能看到：

- 连接时一次 `WriteSingleCoil(0, true)`
- 退出时连续两次 `WriteSingleCoil(0, false)`
- 随后出现 `timed out writing Zhongsheng relay coil`

也就是说：

- 不是没有断电逻辑
- 而是同一块板在退出时被并发/重复断电

## 重构前后设计变化

### 旧设计的问题

- board 是否可用，依赖 `sessions` map 是否存在
- websocket cleanup 和 `DELETE session` 都能单独决定资源释放
- `session` 生命周期与资源释放顺序分散在多个地方
- 最终导致同一 board 的 power-off 被打两次

### 新设计

本次重构采用：

- `session actor` 负责 session 生命周期
- `ReleaseCoordinator` 负责统一资源释放
- board 运行时状态显式化：
  - `idle`
  - `using`
  - `releasing`
  - `error`
- `Drop` 不直接做异步释放，只做 best-effort 停止通知

统一释放顺序：

1. session actor 收到 stop 原因
2. session 状态切为 `releasing`
3. board 状态切为 `releasing`
4. ReleaseCoordinator 执行一次且仅一次：
   - 停止相关任务
   - `power-off`
   - 清理 TFTP/session 文件
5. 成功则 board 回到 `idle`
6. 失败则 board 进入 `error`

### API 语义变化

- `DELETE /api/v1/sessions/{id}` 变为“发起释放”，返回 `202 Accepted`
- session 对外暴露 `state`
- 新增 board runtime 状态查询接口，便于排查：
  - `lease_state`
  - `active_session_id`
  - `last_release_error`
  - `updated_at`

## 本次实现后要特别注意的点

### 1. `ostool` 运行命令要显式指定 binary

因为 `ostool` crate 有多个 binary，连接板子时应使用：

```bash
cargo run -p ostool --bin ostool -- board connect -b orangepi5plus --server 127.0.0.1 --port 2999
```

### 2. `DELETE session` 不再代表“释放已完成”

现在它只代表：

- release 已发起
- 后台正在进行

真正是否释放完成，应结合：

- session 是否从列表消失
- board runtime 是否回到 `idle`
- 是否有 `last_release_error`

### 3. 如果 release 失败，board 会进入 `error`

不会自动重新分配，以避免：

- 板子实际还没断电
- TFTP/session 资源残留
- 串口释放不完整

### 4. websocket 关闭本身不再直接 power-off

它只负责发送 stop 请求。

真正断电只能由统一 releaser 执行一次。

这是本次修复重复 power-off 的关键。

## 最终人工验收流程

本次最终验收使用真实三终端流程：

### 终端 1：启动 server

```bash
RUST_LOG=debug cargo run -p ostool-server -- --config .ostool-server.toml
```

### 终端 2：观察 session 生命周期

```bash
watch -n 0.2 'curl -sS http://127.0.0.1:2999/api/v1/admin/sessions'
```

### 终端 3：连接真实 client

```bash
cargo run -p ostool --bin ostool -- board connect -b orangepi5plus --server 127.0.0.1 --port 2999
```

进入串口后：

- 等待约 4 秒
- 手动按 `Ctrl+A`
- 再按 `x`

### 期望结果

- client 正常退出串口模式
- session 很快从 `/api/v1/admin/sessions` 中消失
- server debug 日志只出现一次有效 `WriteSingleCoil(0, false)`
- 不再出现“双重 power-off 导致 timeout”的 warning

## 本次最终人工验收结果

本次重构后，按上述真实流程复现得到：

- client 正常退出，打印 `Exit by: Ctrl+A+x`
- session 在短时间内消失
- server 日志中只出现一次 release 协调日志

## 后续建议

- 如果后面需要继续定位硬件侧问题，优先保留 `RUST_LOG=debug`
- 若需要更细的人工排查，可新增一个 admin 接口直接查看 board runtime 状态
- 若以后引入更多资源类型，也统一挂到 `ReleaseCoordinator`，不要再让多个路径各自释放
